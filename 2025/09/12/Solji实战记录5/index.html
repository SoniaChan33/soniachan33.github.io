SoniaChen2025-09-122025-09-13

Solji实战记录5_基础捐助功能和动态配置优化

基础捐助功能

需求

  • 用户可向寺庙直接捐款（SOL、SOLJI或USDC，SPL-token）
  • 捐助会记录每个人的捐助金额，大于0.5sol可获得免费mint 佛像的机会，截止日期后按捐助多少排行的前1万名进行分发佛像NFT
  • 每次捐助可以获得「寺庙勋章 NFT，可持续升级」
捐助等级 最低金额（SOL） 勋章名称（NFT，可升级） 用户功德奖励 特别权益 寺庙香火值
铜牌信士 0.05 入门功德铜章 +65 点亮香火墙名字 +1200
银牌居士 0.2 精进银章 +1300 可为寺庙投票提案 +6300
金牌护法 1.0 护法金章 +14000 可参与寺庙 NFT 治理 +30000
至尊供奉 5.0 至尊龙章 NFT +120000 解锁彩蛋内容+寺庙共建者身份 +100000

功能要点：

  • 捐助合约支持收款、记录、NFT 分发及功德统计。
  • 可设置寺庙「升级进度条」，捐助是推动升级的一种方式。

捐助后香型解锁更多等级（如“秘制香”、“天界香”）。 =》隐藏玩法，不可兼得

捐助金额大于5Sol可获得秘制香（每5S获得10根，低于50之前，可每次获得10根）；

大于50Sol获得天界香5根，50的倍数，100获得10个，上不封顶。

每捐助0.01sol ，可以增加烧香的1次=》烧香部分每天重置，等级按照上面的规则不变。

开发计划

  • 基础捐助功能
    • 创建捐助指令（支持SOL）
    • 捐助记录数据结构
    • 等级计算逻辑
    • 基础奖励系统
  • NFT勋章系统
    • 勋章NFT数据结构
    • NFT铸造逻辑
    • 勋章升级机制
  • 高级功能
    • SPL代币支持
    • 佛像NFT分发系统
    • 香型解锁逻辑
    • 排名和截止日期系统

  • ✅ 烧香次数增加（每0.01 SOL增加1次）
  • ✅ 佛像NFT免费mint资格（≥0.5 SOL）

数据结构

  • ✅ 用户状态扩展（捐助金额、等级、统计）
  • ✅ 等级计算逻辑
  • ✅ 奖励分发系统

测试

  1. ✅ 铜牌信士等级捐助 - 0.05 SOL捐助获得铜牌等级和奖励
  2. ✅ 佛像NFT资格 - 0.5 SOL捐助获得免费mint佛像资格
  3. ✅ 累积捐助 - 多次捐助正确累积金额和统计

核心数据结构

1. 香型配置 (IncenseTypeConfig)

#[account]
#[derive(InitSpace)]
pub struct IncenseTypeConfig {
    pub incense_id: u8,
    pub name: String,           // 香型名称
    pub price_lamports: u64,    // 价格
    pub merit_reward: u64,      // 功德奖励
    pub incense_points: u64,    // 香火值奖励
    pub is_donation_only: bool, // 是否仅捐助解锁
    pub unlock_threshold: u64,  // 解锁门槛(捐助金额)
    pub is_active: bool,        // 是否启用
}

2. 捐助等级配置 (DonationLevelConfig)

#[account]
#[derive(InitSpace)]
pub struct DonationLevelConfig {
    pub level: u8,              // 等级ID
    pub name: String,           // 等级名称
    pub min_amount: u64,        // 最低金额( lamports)
    pub merit_reward: u64,      // 功德奖励
    pub incense_reward: u64,    // 香火值奖励
    pub special_privileges: Vec<String>, // 特殊权益
    pub is_active: bool,        // 是否启用
}

3. 抽签概率配置 (FortuneProbabilityConfig)

#[account]
#[derive(InitSpace)]
pub struct FortuneProbabilityConfig {
    pub user_type: u8,          // 0:普通用户, 1:佛像持有者
    pub great_luck_prob: u8,    // 大吉概率(0-100)
    pub good_luck_prob: u8,     // 中吉概率
    pub neutral_prob: u8,       // 平概率
    pub bad_luck_prob: u8,      // 凶概率
    pub great_bad_luck_prob: u8, // 大凶概率
}

4. 系统限制配置 (SystemLimitsConfig)

#[account]
#[derive(InitSpace)]
pub struct SystemLimitsConfig {
    pub daily_burn_limit: u8,           // 每日烧香次数限制
    pub daily_free_draw_limit: u8,      // 每日免费抽签次数
    pub daily_free_wish_limit: u8,      // 每日免费许愿次数
    pub buddha_mint_threshold: u64,     // 佛像mint资格门槛
    pub merit_per_draw: u64,            // 抽签消耗功德
    pub merit_per_extra_wish: u64,      // 额外许愿消耗功德
}

管理指令设计

Admin指令列表

  1. update_incense_type - 更新香型配置
  2. add_donation_level - 添加捐助等级
  3. update_donation_level - 更新捐助等级
  4. update_fortune_probabilities - 更新抽签概率
  5. update_system_limits - 更新系统限制
  6. toggle_incense_type - 启用/禁用香型
  7. toggle_donation_level - 启用/禁用捐助等级

实现优先级

第一阶段：基础配置

  1. ✅ 香型配置动态化 (最重要)
  2. ✅ 捐助等级配置动态化
  3. ✅ 系统限制配置动态化

第二阶段：高级配置

  1. ⏳ 抽签概率配置动态化
  2. ⏳ 特殊事件配置
  3. ⏳ 季节性活动配置

技术实现要点

存储设计

  • 使用PDA存储配置数据
  • 支持配置版本控制
  • 提供配置迁移机制

权限控制

  • 只有管理员可以修改配置
  • 敏感配置需要多签确认
  • 配置变更需要审核机制

缓存优化

  • 配置数据缓存到程序内存
  • 支持配置热更新
  • 提供配置快照功能

你希望我从哪个配置开始实现动态化？我建议先从香型配置开始，因为这是最基础也是最常调整的配置。
