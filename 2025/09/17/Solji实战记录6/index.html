预言机本地测试问题总结

问题背景

在Solana程序开发中，预言机（Oracle）是获取外部数据的重要工具，但本地测试环境通常没有真实的预言机服务，导致测试失败。

原始错误

Error: Account `randomnessAccount` not provided.
AnchorError caused by account: randomness_account. Error Code: AccountNotEnoughKeys.

最佳解决方案：条件编译 + 特征标志

1. Cargo.toml - 添加特征标志
[features]
default = []
cpi = ["no-entrypoint"]
no-entrypoint = []
no-idl = []
no-log-ix-name = []
idl-build = ["anchor-lang/idl-build", "anchor-spl/idl-build"]
devnet = []
localnet = []  # 🆕 没有的话 新增
2. Rust程序 - 条件编译
文件：anchor/programs/temple/src/instructions/draw_fortune.rs
#[derive(Accounts)]
pub struct DrawFortune<'info> {
    #[account(mut)]
    pub user: Signer<'info>,
    
    #[account(
        mut,
        seeds = [UserState::SEED_PREFIX.as_bytes(), user.key().as_ref()],
        bump = user_state.bump,
    )]
    pub user_state: Account<'info, UserState>,
    
    #[account(
        seeds = [TempleConfig::SEED_PREFIX.as_bytes()],
        bump,
    )]
    pub temple_config: Account<'info, TempleConfig>,
    
    /// CHECK: 随机数账户（仅在非mock模式下需要）
    #[cfg(not(feature = "localnet"))]  # 🆕 条件编译
    pub randomness_account: AccountInfo<'info>,
}
// 生成随机数：根据编译特征决定使用哪种方式
#[cfg(feature = "localnet")]
let random_value = {
    // 使用系统时钟作为伪随机数种子（测试模式）
    let clock = Clock::get()?;
    let seed = clock.unix_timestamp as u64 + clock.slot;
    (seed % 100) as u8
};

#[cfg(not(feature = "localnet"))]
let random_value = {
    // 使用真实预言机（生产模式）
    let clock = Clock::get()?;
    let randomness_data = switchboard_on_demand::RandomnessAccountData::parse(
        ctx.accounts.randomness_account.data.borrow(),
    )
    .map_err(|_| ErrorCode::InvalidRandomness)?;
    
    let revealed_random_value = randomness_data
        .get_value(clock.slot)
        .map_err(|_| ErrorCode::RandomnessNotResolved)?;
    
    let mut random_bytes = [0u8; 8];
    random_bytes.copy_from_slice(&revealed_random_value[..8]);
    let random_u64 = u64::from_le_bytes(random_bytes);
    (random_u64 % 100) as u8
};
3. 测试代码 - 移除账户依赖

文件：anchor/tests/utils/setup.ts

public async drawFortune(user: Keypair, useMerit: boolean = false): Promise<any> {
    const [userStatePda] = PublicKey.findProgramAddressSync(
        [Buffer.from("user_state"), user.publicKey.toBuffer()],
        this.program.programId
    );

    // For local testing with localnet, we don't need randomnessAccount
    const tx = await this.program.methods
        .drawFortune(useMerit)
        .accounts({
            user: user.publicKey,
            userState: userStatePda,
            templeConfig: this.templeConfigPda,
            // randomnessAccount is not needed in localnet mode
        })
        .signers([user])
        .rpc();

    return tx;
}
4. 构建脚本 - 自动化特征选择

文件：package.json

{
  "scripts": {
    "test:all": "cd anchor && anchor clean &&  anchor build -- --features localnet && anchor deploy && cd anchor && ANCHOR_PROVIDER_URL=http://localhost:8899 ANCHOR_WALLET=~/.config/solana/id.json vitest run"
  }
}

本地开发/测试

# 清理缓存
anchor clean

# 使用mock特征构建（用于测试）
anchor build -- --features localnet

# 部署到本地网络
anchor deploy

# 运行测试
npm test

生产部署

anchor build      # 使用真实预言机
anchor deploy

捐助模块 NFT BUG与解决方案

问题1：生命周期参数不兼容

问题表现：

fn mint_new_medal_nft<'a>(
    donor: &Signer,
    user_state: &mut Account<UserState>,
    medal_nft_account: &mut Account<MedalNFT>,
    temple_config: &Account<TempleConfig>,  // 缺少生命周期标注
    donation_amount: u64,
    nft_mint_account: AccountInfo<'a>,  // 这里有生命周期
) -> Result<()>

解决方案：

fn mint_new_medal_nft<'a>(
    donor: &Signer<'a>,                     // 添加生命周期
    user_state: &mut Account<'a, UserState>, // 添加生命周期
    medal_nft_account: &mut Account<'a, MedalNFT>, // 添加生命周期
    temple_config: &Account<'a, TempleConfig>, // 添加生命周期
    donation_amount: u64,
    nft_mint_account: AccountInfo<'a>,      // 保持一致生命周期
    nft_associated_token_account: AccountInfo<'a>,
    meta_account: AccountInfo<'a>,
    token_program: AccountInfo<'a>,
    system_program: AccountInfo<'a>,
    rent: AccountInfo<'a>,
    token_metadata_program: AccountInfo<'a>,
) -> Result<()>

原理：所有相关的账户参数必须共享同一生命周期，以确保借用检查器可以验证引用关系的有效性。

问题2：AccountInfo 的移动语义冲突

问题表现：

anchor_spl::token::freeze_account(CpiContext::new_with_signer(
    token_program,
    FreezeAccount { ... },
    &[...]
))?;  // 第一次使用 token_program

// 稍后再次使用 token_program 会失败
mint_to(CpiContext::new(...))?;

解决方案：显式克隆所有会被多次使用的参数

let meta_account = meta_account.clone();
let system_program = system_program.clone();
let rent = rent.clone();
let token_program_clone = token_program.clone();

原理：AccountInfo 的 borrow checker 默认应用移动语义，就像 std::marker::Copy 没有实现一样。

问题4：不存在的方法调用

问题表现：

let temple_signer_seeds: &[&[&[u8]]] = &[&[
    TempleConfig::SEED_PREFIX.as_bytes(),
    &[temple_config.bump()], // 方法不存在
]];

解决方案：直接在指令上下文中使用账户，避免签名器派生

// 移除签名器种子参数，直接使用
CpiContext::new(token_metadata_program, CreateMetadataAccountsV3 {
    metadata: meta_account,
    mint: nft_mint_account,
    mint_authority: temple_config.to_account_info(),
    update_authority: donor.to_account_info(),
    payer: donor.to_account_info(),
    system_program,
    rent,
})

原理：Account 的 bump 方法在指令上下文中无法直接访问，需要依赖 derive(Accounts) 宏的内置功能。

问题5：更新元数据调用参数错误

问题表现：

update_metadata_accounts_v2(
    CpiContext::new(...),
    None,
    Some(DataV2 { ... }),
    None,
    Some([false; 5]), // 参数类型错误
)?

解决方案：调整到正确的参数顺序

update_metadata_accounts_v2(
    CpiContext::new(token_metadata_program, UpdateMetadataAccountsV2 {
        metadata: meta_account,
        update_authority,
    }),
    None,                    // new_update_authority
    Some(DataV2 { ... }),    // data
    None,                    // primary_sale_happened
    None,                    // is_mutable
)?;

原理：基于实际测试的函数签名，发现参数类型必须完全匹配。

问题7：使用前缀修饰未使用的变量

问题表现：编译器警告未使用的变量参数。

解决方案：添加下划线前缀表示有意未使用

fn upgrade_medal_nft<'a>(
    _donor_pubkey: Pubkey,           // 明确标记不使用
    medal_nft_account: &mut Account<MedalNFT>,
    _donation_amount: u64,          // 明确标记不使用
    _temple_config: &Account<TempleConfig>,  // 明确标记不使用
    ...
)

原理：告诉编译器这些参数是必需的接口设计的一部分，即使当前实现不使用它们。

其他问题

  1. 堆栈溢出 - 函数堆栈偏移超过4096字节限制
  2. 借用检查错误 - 多个可变借用冲突
  3. 账户验证错误 - UserState初始化不完整
  4. 资金不足错误 - 测试账户SOL不足
  5. PDA种子不匹配 - medalNftPda种子缺少"account"部分
  6. 账户获取低效 - 使用getProgramAccounts而不是直接PDA计算

解决方案：

  1. 使用Box包装大型账户，减少堆栈使用
  2. 简化函数结构，将所有NFT逻辑放在donate函数内部
  3. 在init_user.rs中添加所有UserState字段初始化
  4. 增加测试账户空投金额到6 SOL
  5. 修复medalNftPda种子，添加Buffer.from(“account”)
  6. 使用PublicKey.findProgramAddressSync直接计算PDA地址
